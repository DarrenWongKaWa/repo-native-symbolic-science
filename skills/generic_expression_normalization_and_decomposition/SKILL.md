# Generic Expression Normalization and Decomposition

## Purpose
Perform **deterministic structural normalization** and **exact parent-child decomposition** of a raw (or previously normalized) expression. This skill applies only structurally defined, reversible operations: dummy-index canonicalization, collision detection and resolution, deterministic term ordering, exact rational coefficient normalization, function-head normalization, and held-sum normalization. It then registers the normalized expression as a parent and generates child sub-expressions with full lineage tracking. The acceptance gate is EXACT parent reconstruction from children — not term-count equality, not heuristic similarity.

## Activation Conditions
This skill MUST be activated when:
- The routing skill emits `routing_target: "generic_expression_normalization_and_decomposition"` with `task_class: "transformation"` and `sub_class: "decompose"`.
- A raw ingested expression (`RAW_INGESTED`) needs to be structurally normalized before any further transformation.
- A human requests "normalize this expression" or "decompose this expression."
- An adapter specifies that a sector decomposition must be applied to the expression.
- A downstream skill (transformation, verification) requires normalized sub-expressions with deterministic IDs.

This skill MUST NOT be activated for:
- Expressions that are already normalized and have a parent_id (re-normalizing would break lineage unless the previous normalization was found to be incorrect).
- Expressing that are labelled `CANDIDATE_TRANSFORMED` or `VERIFIED` (decomposition is part of the normalization pipeline, not post-verification).
- Algebraic transformations (those belong to `candidate_symbolic_transformation`).

## Required Inputs
1. **Raw object reference** (mandatory): Either a `generic_raw_object.json` reference (for first normalization) or a parent expression from a prior normalization run.
2. **Routing decision** (mandatory): The `routing_decision.json` that authorized this normalization.
3. **Adapter configuration** (optional but strongly recommended):
   - Sector definitions for tagging sub-expressions.
   - Index convention maps for canonicalization.
   - Specific term-ordering preferences.
   - Known collision-reserved dummy-index names.
4. **Transformation constraints** (mandatory): From the raw object's `transformation_constraints` field — these define what downstream transformations are allowed and which regression targets must be preserved.
5. **Prior normalization record** (conditional): If this is a re-normalization, the prior normalization's output to compare against.

## Required Output Directory

Where `{parent_id}` is a deterministic SHA-based identifier derived from the normalized expression and its raw source.

## Allowed Operations
The following structural (non-algebraic) operations are EXPLICITLY ALLOWED:

### Dummy-Index Canonicalization
- Rename dummy (summed) indices to a deterministic canonical naming scheme (e.g., `d1`, `d2`, ... or `α`, `β`, ... by domain).
- The renaming MUST be a bijection — every original dummy index maps to exactly one canonical name, and the mapping is recorded.
- Dummy indices are identified by their role in explicit or implicit sums, NOT by their position.

### Collision Detection and Resolution
- Detect when the same index name appears as both a dummy index (summed) and a free or external index in the same scope.
- Detect when an index name introduced by a transformation would collide with an existing index name.
- Resolve collisions by deterministic renaming of the colliding index (recording the resolution).
- Never rename a free or external index to resolve a collision (the dummy index is always the one renamed).

### Term Ordering
- Apply a deterministic ordering to terms in sums (e.g., lexicographic by canonical string representation, then by coefficient magnitude, then by index structure).
- The ordering MUST be total (every pair of terms has a defined order) and deterministic (same input always produces same output).
- The ordering must be recorded so downstream skills can reproduce it.

### Exact Rational Coefficient Normalization
- Extract coefficients from terms and normalize them to exact rational form (e.g., `1/2` not `0.5`).
- Factor out common factors from all terms in a sum.
- Normalize the overall sign of a term or sum according to a deterministic convention.
- Never approximate a rational coefficient as a floating-point number.

### Function-Head Normalization
- Normalize function heads to a canonical representation (e.g., `Plus[...]` rather than `a + b + c` if the latter is ambiguous).
- Ensure that associative operations are flattened (e.g., `Plus[a, Plus[b, c]]` → `Plus[a, b, c]`).

### Held-Sum Normalization
- Normalize held (unevaluated) sums, products, integrals, and derivatives to a canonical form.
- Ensure that the summation index and bounds are explicitly recorded in a deterministic format.

### Parent Registration
- Register the normalized expression as a "parent" with a deterministic `parent_id`.
- Record the parent-child relationship for all generated children.

### Deterministic Child Generation
- Generate children by extracting sub-expressions at well-defined structural boundaries: terms of a sum, factors of a product, arguments of a function, elements of a list, ranges of a sum/integral, and adapter-supplied sector boundaries.
- Each child receives a deterministic `child_id` derived from its exact content and its parent's `parent_id`.

### Adapter-Supplied Sector Tagging
- Apply sector tags defined by the adapter to appropriate sub-expressions.
- Sector tags propagate to children that inherit their parent's sector membership.

## Forbidden Operations
- **Any algebraic transformation** — this is structural normalization only. Do not apply identities, expand products, contract indices, substitute definitions, or perform any operation that changes the mathematical value of the expression.
- **Term-count-quality-only reconstruction** — do not accept reconstruction that merely has the same number of terms. The reconstructed expression must be EXACTLY equal to the parent.
- **Applying integration-by-parts** — this is a differential identity (Level D/E), not structural.
- **Applying any identity from an external source** — no definitions, no theorems, no assumptions.
- **Changing the mathematical value of the expression** — normalization must be strictly value-preserving.
- **Silent dummy-index collision introduction** — if a collision is detected but cannot be resolved (e.g., all canonical names are exhausted for that domain), escalate rather than proceeding.
- **Overwriting an existing parent** — if a parent with the same `parent_id` already exists, create a new version; do not modify the existing one in place.
- **Guessing index domains** — if an index domain is unknown, escalate via `human_scientist_semantic_escalation`.
- **Applying normalization to a `CANDIDATE_TRANSFORMED` or `VERIFIED` expression without explicit re-normalization authorization.**
- **Claiming `VERIFIED` or `CANONICAL` status** — the output is always `NORMALIZED_PARENT`.

## Semantic Blockers
The following conditions MUST block normalization and require escalation:
1. **Unknown index domain**: An index appears in the expression but its domain (range, type) is not known from the raw object or adapter.
2. **Ambiguous summation convention**: It is unclear whether a repeated index is summed or not.
3. **Unresolvable dummy collision**: A required renaming cannot be done because all valid names in the index domain are exhausted.
4. **Non-associative operator ambiguity**: An operator's associativity is unknown, making flattening/grouping ambiguous.
5. **Branch-sensitive function in sum**: A branch-sensitive function (e.g., `Abs`, `Sign`, `Arg`) appears in a context where normalization might change its evaluation (e.g., inside a sum with indeterminate sign).
6. **Circular parent-child dependency**: The decomposition would create a child that is identical to the parent (infinite recursion risk).
7. **Missing transformation constraints**: The raw object does not specify `allowed_transformations`, so the normalization cannot determine which child boundaries are safe.
8. **Adapter/source convention conflict**: The adapter specifies a sector decomposition that does not match the actual expression structure.

## Task Lifecycle
1. **RECEIVE**: Accept the raw object reference and routing decision.
2. **LOAD_RAW**: Load the `generic_raw_object.json` and `frozen_source_copy`.
3. **LOAD_CONSTRAINTS**: Load `allowed_transformations`, `forbidden_transformations`, and `regression_targets`.
4. **LOAD_ADAPTER**: Load adapter configuration (sector definitions, index conventions, ordering preferences).
5. **VALIDATE_PRE_NORMALIZATION**: Check that preconditions are met (index domains known, summation convention clear, no blockers).
6. **DUMMY_CANONICALIZE**: Rename all dummy indices to canonical names. Resolve collisions. Record mappings.
7. **FLATTEN**: Flatten associative operators (e.g., nested sums).
8. **NORMALIZE_COEFFICIENTS**: Extract and normalize rational coefficients. Factor common factors. Normalize overall sign.
9. **NORMALIZE_FUNCTION_HEADS**: Ensure function heads are in canonical representation.
10. **NORMALIZE_HELD_SUMS**: Normalize held sums, products, integrals, derivatives.
11. **ORDER_TERMS**: Apply deterministic term ordering.
12. **REGISTER_PARENT**: Compute `parent_id`, create `normalized_parent.json`.
13. **DECOMPOSE**: Extract children at structural boundaries. Apply sector tags.
14. **COMPUTE_CHILD_IDS**: Compute deterministic `child_id` for each child.
15. **RECONSTRUCT**: Reconstruct parent from children. Verify EXACT equality.
16. **DECIDE**: If reconstruction is EXACT → write output artifacts and set `status: NORMALIZED_PARENT`. If reconstruction is APPROXIMATE or FAILED → this is a CRITICAL error; halt and escalate.
17. **WRITE**: Produce all output artifacts.
18. **LOG**: Append to the repo-level normalization registry.

## Acceptance Gate: EXACT Parent Reconstruction
The reconstruction gate is the CRITICAL quality check for this skill:
1. The parent expression is serialized in a canonical format.
2. The children are combined according to the `reconstruction_formula`.
3. The combined expression is compared to the parent using `held_expression_subtraction` (literal comparison of held forms) or `term_by_term_comparison`.
4. If the difference is identically zero (or held-form identical), the reconstruction is `EXACT`.
5. If the difference is nonzero but small (e.g., due to floating-point), the reconstruction is `APPROXIMATE` — this is a FAILURE for the gate.
6. If the difference is nonzero and not small, the reconstruction is `FAILED`.
7. **Term-count equality alone is INSUFFICIENT** — two expressions can have the same number of terms and be mathematically different.

If the reconstruction gate fails, the skill MUST:
1. Halt normalization.
2. Record the reconstruction failure with the exact difference in `parent_reconstruction_test.json`.
3. Escalate to `human_scientist_semantic_escalation` with the failure details.
4. NOT produce any child expressions that would be used by downstream skills.

## Relation / Claim Types
This skill produces the following claims:
- `NORMALIZED_PARENT`: The expression has been structurally normalized and is a valid parent for child decomposition.
- `PARENT_CHILD_DECOMPOSITION`: The specified children are an EXACT decomposition of the parent (reconstruction verified).
- `DUMMY_CANONICALIZED`: Dummy indices have been renamed according to the canonical scheme.
- `SECTOR_TAGGED`: Adapter-supplied sector tags have been applied.

This skill does NOT produce:
- `CANDIDATE_TRANSFORMED` (no algebraic transformation was performed).
- `VERIFIED` (no identity was verified).
- `CANONICAL` (no canonical promotion was performed).

## Artifact Contract
- `normalized_parent.json` MUST contain `status: "NORMALIZED_PARENT"`.
- `parent_reconstruction_test.json` MUST have `reconstruction_result: "EXACT"` for the output to be valid.
- `child_expressions.json` MUST NOT be produced if `parent_reconstruction_test.json` has `reconstruction_result` other than `"EXACT"`.
- Every child MUST have all required fields: `child_id`, `parent_id`, `exact_expression`, `transformation_rule`, `source_location`, `index_topology`, `operator_signature`, `coefficient`, `sector_tags`, `sha256`.
- The concatenation of all `child_ids` sorted + `parent_id` MUST be unique within the repo.
- All timestamps MUST be ISO 8601 UTC.
- All normalization rules applied MUST be individually recorded with `rule_reversible: true`.

## Downstream Eligibility
A normalized parent and its children are eligible for downstream processing ONLY if:
1. `normalized_parent.json` exists with `status: "NORMALIZED_PARENT"`.
2. `parent_reconstruction_test.json` shows `reconstruction_result: "EXACT"`.
3. All children in `child_expressions.json` have valid `sha256` fields that match their `exact_expression`.
4. No semantic blockers remain unresolved.
5. The parent has not been superseded by a re-normalization.

## Human Escalation Behavior
- Semantic blockers during normalization are escalated via `human_scientist_semantic_escalation`.
- The escalation MUST specify: the blocked normalization step, the specific index/symbol/operator that is ambiguous, what the adapter says (if anything), what the source says (if anything), and what the human should clarify.
- If the human resolves the semantic gap, normalization resumes from the blocking step.
- If the human provides a revised source, the entire normalization is re-run.
- If the reconstruction gate fails, the escalation MUST include the exact reconstruction difference, so the human can determine if the failure is a bug (in the normalization logic) or a fundamental limitation (e.g., the expression cannot be decomposed into the chosen children).

## Error Handling
- **Unrecognized index domain**: Escalate — do not proceed without knowing the domain.
- **Reconstruction failure**: Critical error. Halt, record the difference, escalate to human.
- **Circular decomposition**: Halt. Escalate with the cycle description.
- **Child_id collision**: Two children with different expressions but the same child_id — this indicates a hash collision or a bug. Halt and escalate.
- **Adapter parse failure**: If the adapter cannot be parsed or applied, proceed without it but flag the missing sector tags.
- **Memory/computation exceeded**: If the expression is too large to normalize within resource limits, record a partial normalization with `status: NORMALIZATION_RESOURCE_EXCEEDED` and escalate with size estimates.
